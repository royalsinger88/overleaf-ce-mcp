# overleaf-ce-mcp 使用手册（详细版）

本文档面向以下目标：
- 在自建 Overleaf CE 上稳定登录、同步、上传、编译 LaTeX 稿件。
- 支持 SCI/OA 论文写作流程（例如 Ocean Engineering）。
- 避免常见坑（认证、误同步、replace 删除失败、.olignore 误排除等）。

配套速查：
- [快速上手（10 条命令）](./快速上手.md)

## 1. 项目能力概览

`overleaf-ce-mcp` 提供两类核心能力：
- 同步能力：基于 `overleaf-sync-ce (ols)` 与 CE 做双向/单向同步。
- 通用上传能力：不依赖老旧 file-tree 细节，直接走 CE `/project/new/upload` 上传 zip。

## 2. 环境准备

### 2.1 基础依赖

建议环境：
- Linux VPS（推荐）
- Python >= 3.11
- 已安装 `zip/unzip`

如需本地编译 LaTeX：

```bash
sudo apt-get update
sudo apt-get install -y texlive-latex-base texlive-latex-extra latexmk
```

### 2.2 安装项目

```bash
cd /root/overleaf-ce-mcp
/root/miniconda3/bin/python -m venv .venv
source .venv/bin/activate
pip install -U pip
pip install .
```

如需单独安装 `ols`：

```bash
pip install overleaf-sync-ce
```

### 2.3 启动 MCP 服务

```bash
source /root/overleaf-ce-mcp/.venv/bin/activate
overleaf-ce-mcp
```

## 3. 认证与 `.olauth`

### 3.1 标准登录（有图形界面）

```bash
ols login --path /root/.olauth --ce-url "http://<CE_HOST>:17880"
```

### 3.2 无图形环境（推荐手工 cookie 方式）

当 VPS 无 GUI 或 `ols login` 不可用时，可用浏览器 cookie 手工生成 `.olauth`。

关键点：
- CE 常用 cookie 名是 `overleaf.sid`（不是必须有 `sharelatex.sid`）。
- 项目已兼容 `overleaf.sid`/`sharelatex.sid` 自动别名。

示例脚本（按你实际 cookie/csrf 替换）：

```bash
python - <<'PY'
import pickle, pathlib
p = pathlib.Path('/root/.olauth')
data = {
    'cookie': {
        'overleaf.sid': 's:xxxx.yyyy',
    },
    'csrf': 'your_csrf_token_if_available'
}
pickle.dump(data, p.open('wb'))
print('written', p)
PY
```

验证认证是否有效：

```bash
python - <<'PY'
from overleaf_ce_mcp.upload import list_projects
ps = list_projects('http://<CE_HOST>:17880', '/root/.olauth')
print('projects', len(ps))
for p in ps[:10]:
    print(p.get('name'), p.get('id'))
PY
```

## 4. 快速开始（推荐流程）

## 4.1 初始化论文模板（Ocean Engineering OA）

调用 MCP 工具 `init_manuscript_from_template`：

- `template_name`: `ocean-engineering-oa`
- `target_dir`: 目标空目录（必须）
- 可选：`title`、`authors`、`corresponding_email`、`keywords`
- 可选：`init_paper_state`（默认 `true`）

默认会创建 `paper_state` 目录，用于沉淀输入素材、中间产物、复盘和证据账本。

目录示例：

```text
/root/overleaf-workspace/OE_OA_FirstDraft_20260228
```

## 4.2 本地写作

用 `write_file` 或你自己的编辑器维护：
- `main.tex`
- `sections/*.tex`
- `references.bib`
- `paper_state/*`（建议把检索、实验、复盘持续写入）

## 4.2.1 对已有项目补建 `paper_state`

调用 `init_paper_state_workspace`：
- `project_dir`（必填）
- 可选：`title`、`authors`、`corresponding_email`、`keywords`
- 可选：`force`（默认 `false`，存在文件不覆盖）

## 4.3 本地编译检查

使用 `compile_latex`：
- `project_dir`: 稿件目录
- `main_tex`: 默认 `main.tex`

## 4.4 上传到 CE

推荐使用 `upload_project_dir`，按场景选择：

1) 创建新项目（默认）
- 不传 `target_project`
- 自动：目录打包 -> 上传 -> 可选健康检查

2) 更新已有项目
- 传 `target_project`
- `existing_project_strategy=merge`：合并更新，不删远端多余文件
- `existing_project_strategy=replace`：远端按本地覆盖（会删除远端多余文件）

3) 先预演不执行
- `dry_run=true`

## 5. 关键工具详解

## 5.1 `upload_project_dir`

主要参数：
- `ce_url`：例如 `http://129.150.32.65:17880`
- `project_dir`：本地稿件目录
- `store_path`：默认 `~/.olauth`
- `exclude_globs`：数组，追加排除规则
- `dry_run`：只预览执行计划
- `target_project`：已有项目名（传了就走“已有项目模式”）
- `existing_project_strategy`：`merge`/`replace`
- `health_check`：默认 `true`
- `compile_check`：默认 `true`
- `verbose`：同步日志

默认排除规则已包含：
- `.git/**`
- `__pycache__/**`
- `*.aux *.log *.pdf *.zip ...`

### 示例 A：新建项目上传

```json
{
  "ce_url": "http://129.150.32.65:17880",
  "project_dir": "/root/overleaf-workspace/OE_OA_FirstDraft_20260228",
  "health_check": true,
  "compile_check": true
}
```

### 示例 B：已有项目 merge

```json
{
  "ce_url": "http://129.150.32.65:17880",
  "project_dir": "/root/overleaf-workspace/test",
  "target_project": "test",
  "existing_project_strategy": "merge",
  "health_check": true,
  "compile_check": true
}
```

### 示例 C：已有项目 replace（覆盖）

```json
{
  "ce_url": "http://129.150.32.65:17880",
  "project_dir": "/root/overleaf-workspace/test",
  "target_project": "test",
  "existing_project_strategy": "replace",
  "health_check": true,
  "compile_check": true
}
```

### 示例 D：dry-run

```json
{
  "ce_url": "http://129.150.32.65:17880",
  "project_dir": "/root/overleaf-workspace/test",
  "target_project": "test",
  "existing_project_strategy": "replace",
  "dry_run": true
}
```

模式：

- `local-only`
- `remote-only`

参数：

- `project_name`
- `ce_url`
- `store_path`
- `olignore`
- `delete_policy`：`d/r/i`

当同时传：
- `workspace_path=/root/overleaf-workspace`
- `project_name=test`

且存在 `/root/overleaf-workspace/test` 目录时，系统会自动将同步路径收敛为该子目录，避免误同步整个 workspace。

## 5.3.1 `init_paper_state_workspace`

作用：
- 在已有项目中补建标准化 `paper_state` 工作区。
- 用于沉淀检索输入、实验中间产物、每日/每周复盘和证据账本。

参数：
- `project_dir`（必填）
- `title` / `authors` / `corresponding_email` / `keywords`（可选）
- `force`（默认 `false`）

主要生成结构：
- `paper_state/inputs/literature/seed_queries.yaml`
- `paper_state/inputs/literature/reading_queue.csv`
- `paper_state/inputs/literature/refs_raw.bib`
- `paper_state/inputs/loop.yaml`
- `paper_state/review/daily/`
- `paper_state/review/weekly/`
- `paper_state/memory/claim_evidence.jsonl`

## 5.10.1 `run_optimization_loop`

用途：
- 执行“受控循环优化”（拉尔夫循环）：R1/R2 迭代，自动评分与停止。

默认读取：
- `paper_state/inputs/loop.yaml`（若存在）

核心停止条件：
- 达到 `target_score`
- 连续 `patience` 轮提升小于 `min_score_improvement`
- 连续 `patience` 轮无新增证据

主要产物：
- `paper_state/outputs/optimization_loop/round_XX/round_result.json`
- `paper_state/outputs/optimization_loop/round_XX/round_summary.md`
- `paper_state/outputs/optimization_loop/loop_summary.json`
- `paper_state/outputs/optimization_loop/loop_summary.md`
- `paper_state/review/daily/YYYY-MM-DD.md`（可选）

## 5.10.2 `run_paper_cycle`

用途：
- 低摩擦一键执行：优化循环 + 日报 +（按策略）周报。
- 默认从 `paper_state/inputs` 自动取配置，减少每轮手工拼参数。

参数：
- `project_dir`（必填）
- `day`（可选，`YYYY-MM-DD`）
- `weekly_mode`：`auto` / `always` / `never`（默认 `auto`，周五触发周报）
- `run_loop`（默认 `true`）
- `run_daily`（默认 `true`）
- `overwrite_reviews`（默认 `true`）
- `write_state`（默认 `true`）
- `auto_scan_inputs`（默认 `true`）：自动扫描 `paper_state/inputs` 并补齐缺省参数
- `use_cache`（默认 `true`）：启用增量缓存
- `cache_ttl_hours`（默认 `24`）：缓存有效期
- `force_refresh`（默认 `false`）：忽略缓存强制重算
- `write_missing_checklist`（默认 `true`）：缺失项写入 `paper_state/inputs/INPUT_MISSING.md`
- `strict_missing`（默认 `false`）：若有缺失素材则直接报错停止
- 其余参数与 `run_optimization_loop` 一致（用于覆盖默认输入）

产物：
- `paper_state/outputs/paper_cycle/cycle_*.json`
- `paper_state/outputs/paper_cycle/cycle_*.md`
- `paper_state/outputs/paper_cycle/material_scan_*.json`
- `paper_state/outputs/paper_cycle/material_scan_*.md`
- `paper_state/inputs/INPUT_MISSING.md`（存在缺失时）
- `paper_state/cache/paper_cycle/*.json`（启用缓存时）
- 日报/周报与状态回写（按执行策略）

## 5.10.3 `run_paper_doctor`

用途：
- 对 `paper_state/inputs` 做格式与完整性检查，输出可修复建议。

## 5.10.4 `list_upgrade_tasks` / `run_priority_upgrade_loop`

用途：
- 将“六项改造方向”做成可循环执行的优先级任务队列。
- 支持 `dry_run`、`max_tasks`、`resume`（断点续跑）。
- 状态文件：`paper_state/memory/upgrade_loop_state.json`。

## 5.10.5 `list_generic_priority_tasks` / `run_generic_priority_loop`

用途：
- 将“优先级循环”抽象为通用能力，适用于非论文项目。
- 通过 `plan.json` 或 `task_text` 定义任务，支持 `noop` 与 `shell` 两类 action。

关键参数：
- `workspace_dir`、`plan_path`（可选）
- `task_text`（可选，与 `plan_path` 二选一）
- `dry_run`、`resume`
- `allow_shell`（默认 false，执行 shell 任务时需显式开启）
- `command_template`（`task_text` 场景可选，支持 `{task}` 占位符）

输出目录：
- `.codex/priority-loop/state.json`
- `.codex/priority-loop/runs/run_*.json`

补充：
- 可先调用 `list_generic_priority_plan_templates` 查看内置模板。
- 用 `init_generic_priority_plan` 将模板落地为当前项目 `plan.json`。

## 5.5 `search_academic_papers`

作用：
- 使用官方 API 检索论文元数据，支持 `arXiv`、`OpenAlex`、`Crossref`、`Semantic Scholar`、`OpenReview`。

参数：
- `query`（必填）
- `source`：`all` / `arxiv` / `openalex` / `crossref` / `semantic_scholar` / `openreview`
- `max_results_per_source`（默认 8）
- `timeout`（默认 30）
- `s2_api_key`（可选，仅当你要使用 `semantic_scholar` 时需要）

说明：
- `source=all` 默认走无 Key 组合：`arXiv + OpenAlex + Crossref`。
- 若配置了 `S2_API_KEY`（或显式传 `s2_api_key`），`source=all` 会额外叠加 `Semantic Scholar`。
- `OpenReview` 默认不并入 `source=all`，需显式传 `source=openreview`，或使用独立工具 `search_openreview_papers`。
- 当某一源失败时，其他源结果仍会返回。

## 5.5.1 `list_academic_source_capabilities`

作用：
- 返回检索适配器清单与能力标签，便于上层 workflow 自动编排。
- 能直接看到哪些源已启用、哪些源因 key 缺失被跳过。

参数：
- `s2_api_key`（可选）

## 5.5.1.1 `search_openreview_papers`

作用：
- 显式检索 OpenReview 顶会论文（当前支持 `ICLR` / `NeurIPS` / `ICML`）。

参数：
- `query`（必填）
- `venue`（默认 `ICLR.cc`，支持 `ICLR/NeurIPS/ICML` 或对应 `*.cc`）
- `year`（可选，默认上一年）
- `limit`（默认 20，最大 100）
- `timeout`（默认 30）

## 5.5.2 `fetch_paper_fulltext`

作用：
- 获取“可用正文文本”，在全文不可得时自动回退到摘要/元数据。

回退链：
- `Unpaywall` -> `OpenAlex DOI` -> `Crossref DOI` -> `arXiv` -> `直接 URL` -> `标题检索摘要`

参数：
- `title`（可选）
- `doi`（可选）
- `arxiv_id`（可选）
- `url`（可选）
- `unpaywall_email`（可选，不传则读 `UNPAYWALL_EMAIL`）
- `s2_api_key`（可选，仅标题检索回退时可能用到）
- `timeout`（默认 30）

## 5.6.3 `recommend_target_journals`

作用：

## 5.6.4 `verify_reference`

作用：
- 核验参考文献真伪与匹配度，并返回修正后的 BibTeX 草稿。
- 可用于审查 AI 生成引用，减少“编造引用”风险。

参数：
- `title`（可选，`title`/`doi` 至少提供一个）
- `authors`（可选）
- `year`（可选）
- `doi`（可选，`title`/`doi` 至少提供一个）
- `venue`（可选）
- `timeout`（默认 30）
- `s2_api_key`（可选）

## 5.6.5 `sync_zotero_paper_state`

作用：
- 将 `paper_state/inputs/literature` 与 Zotero 做双向同步。
- 支持先 `dry_run` 预演，确认后再执行真实写入。

参数：
- `project_dir`（必填）
- `direction`：`pull` / `push` / `bidirectional`
- `library_id`（可选，不传读取 `ZOTERO_LIBRARY_ID`）
- `api_key`（可选，不传读取 `ZOTERO_API_KEY`）
- `library_type`：`user` / `group`（默认 `user`）
- `limit`（默认 50）
- `query`（可选，pull 时用于 Zotero 端过滤）
- `timeout`（默认 30）
- `dry_run`（默认 `true`）

## 5.7 `generate_deep_research_prompt`

作用：

参数：
- `topic`（必填）
- `known_data`（必填）
- `writing_direction`（必填）
- `core_ideas`（可选）
- `target_journal`（可选）
- `preferred_sources`（可选）
- `output_language`（默认中文）
- `max_references`（默认 30）

## 5.8 `generate_deep_research_prompt_set`

作用：
- 结合 baseline、改进模块、代码资产、实验结果、期刊约束，生成多组深度研究提示词。
- 支持两轮：`r1`（首轮建模）与 `r2`（查漏补缺）。

参数：
- `topic`（必填）
- `known_data`（必填）
- `writing_direction`（必填）
- `baseline_models`（可选）
- `improvement_modules`（可选）
- `code_assets`（可选）
- `experiment_results`（可选）
- `draft_ideas`（可选）
- `target_journal`（可选）
- `constraints`（可选）
- `round_stage`：`r1` / `r2`
- `prior_findings`（r2 推荐）
- `num_prompts`（默认 3，范围 2-6）

## 5.9 `ingest_deep_research_report`

作用：
- 将深度研究报告转成可用于论文写作的参考包：
  - URL 列表
  - DOI / arXiv ID
  - BibTeX 草稿
  - 快速写作备注

参数：
- `report_text` 或 `report_file_path`（至少一个）
- `focus_topic`（可选）
- `max_items`（默认 30）
- `save_reference_note_path`（可选）
- `save_bib_path`（可选）

## 5.10 `synthesize_paper_strategy`

作用：
- 综合多轮研究报告和实验信息，输出：
  - 候选标题
  - 创新点候选
  - 写作侧重点
  - 主张边界与下一步动作

参数：
- `topic`（必填）
- `target_journal`（可选）
- `baseline_models`（可选）
- `improvement_modules`（可选）
- `key_results`（可选）
- `report_summaries`（可选）
- `constraints`（可选）
- `candidate_title_count`（默认 6，范围 3-10）

## 5.11 `init_model_diagram_pack`

作用：
- 一键生成“模型结构图生产包”，用于你说的稳定流程：

## 7. 常见问题排查

## 7.1 `ols: command not found`

原因：脚本目录不在 PATH，或 venv 未激活。  
处理：
- 激活 venv：`source /root/overleaf-ce-mcp/.venv/bin/activate`
- 或显式执行 `ols` 绝对路径

## 7.2 `ImportError: QtCore`（Windows 常见）

原因：`ols login` 依赖 GUI 组件。  
处理：在无 GUI 环境使用手工 `.olauth` 方案。

## 7.3 访问 `/project` 被重定向到 `/login`

原因：cookie 无效或过期。  
处理：重新获取浏览器 cookie 并更新 `.olauth`。

## 7.4 编译返回 200 但仍失败

这是正常现象，需看 `compile_status`：
- `status_code=200` 不代表成功
- 以 `compile_status == success` 为准

## 7.5 replace 模式删不净子目录文件

已修复：支持多级路径删除。若旧环境仍异常，执行：
- `apply_compat_patches`
- 重新同步

## 7.6 误把整个 workspace 同步到单项目

已增加护栏自动收敛路径。仍建议你：
- 明确设置 `project_name`
- 工作区保持“一个项目一个子目录”结构

## 8. Ocean Engineering OA 推荐实战流程

1. 用模板初始化目录。  
2. 用 draw.io MCP 先确定模型结构真值图（`.drawio`）。  
3. 用 `init_model_diagram_pack` + `drawio_file_path` 生成 Nano Banana Pro 提示词包。  
4. 用 `generate_deep_research_prompt_set(round_stage=r1)` 生成第一轮多组提示词并在 GPT 网页版运行。  
5. 用 `ingest_deep_research_report` 解析 R1 报告，提取 DOI/arXiv/BibTeX 草稿。  
6. 用 `build_related_work_pack` 拉取候选论文与补充 BibTeX 草稿。  
7. 如仍有不确定点，用 `generate_deep_research_prompt_set(round_stage=r2)` 做二轮深挖，并再次解析。  
8. 用 `synthesize_paper_strategy` 输出题目候选、创新点和写作侧重点。  
9. 将 BibTeX 合并到 `references.bib`，把报告摘要与 `note_markdown` 作为引言/相关工作素材。  
10. 先本地 `compile_latex`，确保主结构可编译。  
11. 用 `upload_project_dir` + `dry_run=true` 检查打包计划。  
12. 首次上传不传 `target_project`（创建新项目）。  
13. 后续迭代使用 `target_project + merge`。  
14. 需要远端完全对齐时，用 `target_project + replace`。  
15. 每次关键更新后执行 `health_check_project`。
