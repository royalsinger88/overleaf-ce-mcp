# overleaf-ce-mcp 使用手册（详细版）

本文档面向以下目标：
- 在自建 Overleaf CE 上稳定登录、同步、上传、编译 LaTeX 稿件。
- 支持 SCI/OA 论文写作流程（例如 Ocean Engineering）。
- 避免常见坑（认证、误同步、replace 删除失败、.olignore 误排除等）。

配套速查：
- [快速上手（10 条命令）](./快速上手.md)

## 1. 项目能力概览

`overleaf-ce-mcp` 提供两类核心能力：
- 同步能力：基于 `overleaf-sync-ce (ols)` 与 CE 做双向/单向同步。
- 通用上传能力：不依赖老旧 file-tree 细节，直接走 CE `/project/new/upload` 上传 zip。

当前支持的关键工具：
- `check_environment`
- `ols_login`
- `ols_list_projects`
- `ols_sync`
- `init_manuscript_from_template`
- `write_file`
- `compile_latex`
- `package_project_zip`
- `upload_project_zip`
- `upload_project_dir`
- `health_check_project`
- `apply_compat_patches`
- `search_academic_papers`
- `build_related_work_pack`
- `generate_deep_research_prompt`
- `generate_deep_research_prompt_set`
- `ingest_deep_research_report`
- `synthesize_paper_strategy`
- `init_model_diagram_pack`

## 2. 环境准备

### 2.1 基础依赖

建议环境：
- Linux VPS（推荐）
- Python >= 3.11
- 已安装 `zip/unzip`

如需本地编译 LaTeX：

```bash
sudo apt-get update
sudo apt-get install -y texlive-latex-base texlive-latex-extra latexmk
```

### 2.2 安装项目

```bash
cd /root/overleaf-ce-mcp
/root/miniconda3/bin/python -m venv .venv
source .venv/bin/activate
pip install -U pip
pip install .
```

如需单独安装 `ols`：

```bash
pip install overleaf-sync-ce
```

### 2.3 启动 MCP 服务

```bash
source /root/overleaf-ce-mcp/.venv/bin/activate
overleaf-ce-mcp
```

## 3. 认证与 `.olauth`

### 3.1 标准登录（有图形界面）

```bash
ols login --path /root/.olauth --ce-url "http://<CE_HOST>:17880"
```

### 3.2 无图形环境（推荐手工 cookie 方式）

当 VPS 无 GUI 或 `ols login` 不可用时，可用浏览器 cookie 手工生成 `.olauth`。

关键点：
- CE 常用 cookie 名是 `overleaf.sid`（不是必须有 `sharelatex.sid`）。
- 项目已兼容 `overleaf.sid`/`sharelatex.sid` 自动别名。

示例脚本（按你实际 cookie/csrf 替换）：

```bash
python - <<'PY'
import pickle, pathlib
p = pathlib.Path('/root/.olauth')
data = {
    'cookie': {
        'overleaf.sid': 's:xxxx.yyyy',
    },
    'csrf': 'your_csrf_token_if_available'
}
pickle.dump(data, p.open('wb'))
print('written', p)
PY
```

验证认证是否有效：

```bash
python - <<'PY'
from overleaf_ce_mcp.upload import list_projects
ps = list_projects('http://<CE_HOST>:17880', '/root/.olauth')
print('projects', len(ps))
for p in ps[:10]:
    print(p.get('name'), p.get('id'))
PY
```

## 4. 快速开始（推荐流程）

## 4.1 初始化论文模板（Ocean Engineering OA）

调用 MCP 工具 `init_manuscript_from_template`：

- `template_name`: `ocean-engineering-oa`
- `target_dir`: 目标空目录（必须）
- 可选：`title`、`authors`、`corresponding_email`、`keywords`

目录示例：

```text
/root/overleaf-workspace/OE_OA_FirstDraft_20260228
```

## 4.2 本地写作

用 `write_file` 或你自己的编辑器维护：
- `main.tex`
- `sections/*.tex`
- `references.bib`

## 4.3 本地编译检查

使用 `compile_latex`：
- `project_dir`: 稿件目录
- `main_tex`: 默认 `main.tex`

## 4.4 上传到 CE

推荐使用 `upload_project_dir`，按场景选择：

1) 创建新项目（默认）
- 不传 `target_project`
- 自动：目录打包 -> 上传 -> 可选健康检查

2) 更新已有项目
- 传 `target_project`
- `existing_project_strategy=merge`：合并更新，不删远端多余文件
- `existing_project_strategy=replace`：远端按本地覆盖（会删除远端多余文件）

3) 先预演不执行
- `dry_run=true`

## 5. 关键工具详解

## 5.1 `upload_project_dir`

主要参数：
- `ce_url`：例如 `http://129.150.32.65:17880`
- `project_dir`：本地稿件目录
- `store_path`：默认 `~/.olauth`
- `exclude_globs`：数组，追加排除规则
- `dry_run`：只预览执行计划
- `target_project`：已有项目名（传了就走“已有项目模式”）
- `existing_project_strategy`：`merge`/`replace`
- `health_check`：默认 `true`
- `compile_check`：默认 `true`
- `verbose`：同步日志

默认排除规则已包含：
- `.git/**`
- `__pycache__/**`
- `*.aux *.log *.pdf *.zip ...`

### 示例 A：新建项目上传

```json
{
  "ce_url": "http://129.150.32.65:17880",
  "project_dir": "/root/overleaf-workspace/OE_OA_FirstDraft_20260228",
  "health_check": true,
  "compile_check": true
}
```

### 示例 B：已有项目 merge

```json
{
  "ce_url": "http://129.150.32.65:17880",
  "project_dir": "/root/overleaf-workspace/test",
  "target_project": "test",
  "existing_project_strategy": "merge",
  "health_check": true,
  "compile_check": true
}
```

### 示例 C：已有项目 replace（覆盖）

```json
{
  "ce_url": "http://129.150.32.65:17880",
  "project_dir": "/root/overleaf-workspace/test",
  "target_project": "test",
  "existing_project_strategy": "replace",
  "health_check": true,
  "compile_check": true
}
```

### 示例 D：dry-run

```json
{
  "ce_url": "http://129.150.32.65:17880",
  "project_dir": "/root/overleaf-workspace/test",
  "target_project": "test",
  "existing_project_strategy": "replace",
  "dry_run": true
}
```

## 5.2 `upload_project_zip`

用于“你已准备好 zip 文件”的场景，行为是“创建新项目”。

主要参数：
- `ce_url`
- `zip_path`
- `store_path`（可选）
- `timeout`（默认 300）
- `health_check`（默认 true）
- `compile_check`（默认 true）

## 5.3 `ols_sync`

模式：
- `bidirectional`
- `local-only`
- `remote-only`

参数：
- `workspace_path`
- `project_name`
- `ce_url`
- `store_path`
- `olignore`
- `delete_policy`：`d/r/i`
- `verbose`

### 同步安全护栏（已内置）

当同时传：
- `workspace_path=/root/overleaf-workspace`
- `project_name=test`

且存在 `/root/overleaf-workspace/test` 目录时，系统会自动将同步路径收敛为该子目录，避免误同步整个 workspace。

## 5.4 `health_check_project`

参数：
- `ce_url`
- `project_name` 或 `project_id`（至少一个）
- `compile_check`（默认 true）

返回重点字段：
- `ok`
- `compile.compile_status`
- `compile.has_pdf`
- `compile.pdf_url`

## 5.5 `search_academic_papers`

作用：
- 使用官方 API 检索论文元数据，当前支持 `arXiv` 与 `Semantic Scholar`。

参数：
- `query`（必填）
- `source`：`all` / `arxiv` / `semantic_scholar`
- `max_results_per_source`（默认 8）
- `timeout`（默认 30）
- `s2_api_key`（可选，不传则读取环境变量 `S2_API_KEY`）

说明：
- Semantic Scholar 未配置 API key 时可能返回 `429`（限流）。
- 当 `source=all` 且某一源失败时，另一源结果仍会返回。

## 5.6 `build_related_work_pack`

作用：
- 生成“相关工作素材包”，直接服务写作：
  - 候选论文列表
  - `note_markdown`（相关工作草稿骨架）
  - `bibtex_entries`（可粘贴到 `references.bib`）

参数：
- `query`（必填）
- `source`：`all` / `arxiv` / `semantic_scholar`
- `max_results_per_source`（默认 8）
- `max_items_for_note`（默认 8）
- `timeout`（默认 30）
- `s2_api_key`（可选）

## 5.7 `generate_deep_research_prompt`

作用：
- 根据你的已有数据与写作思路，自动生成“可直接粘贴到 GPT 网页版深度研究”的提示词。

参数：
- `topic`（必填）
- `known_data`（必填）
- `writing_direction`（必填）
- `core_ideas`（可选）
- `target_journal`（可选）
- `preferred_sources`（可选）
- `output_language`（默认中文）
- `max_references`（默认 30）

## 5.8 `generate_deep_research_prompt_set`

作用：
- 结合 baseline、改进模块、代码资产、实验结果、期刊约束，生成多组深度研究提示词。
- 支持两轮：`r1`（首轮建模）与 `r2`（查漏补缺）。

参数：
- `topic`（必填）
- `known_data`（必填）
- `writing_direction`（必填）
- `baseline_models`（可选）
- `improvement_modules`（可选）
- `code_assets`（可选）
- `experiment_results`（可选）
- `draft_ideas`（可选）
- `target_journal`（可选）
- `constraints`（可选）
- `round_stage`：`r1` / `r2`
- `prior_findings`（r2 推荐）
- `num_prompts`（默认 3，范围 2-6）

## 5.9 `ingest_deep_research_report`

作用：
- 将深度研究报告转成可用于论文写作的参考包：
  - URL 列表
  - DOI / arXiv ID
  - BibTeX 草稿
  - 快速写作备注

参数：
- `report_text` 或 `report_file_path`（至少一个）
- `focus_topic`（可选）
- `max_items`（默认 30）
- `save_reference_note_path`（可选）
- `save_bib_path`（可选）

## 5.10 `synthesize_paper_strategy`

作用：
- 综合多轮研究报告和实验信息，输出：
  - 候选标题
  - 创新点候选
  - 写作侧重点
  - 主张边界与下一步动作

参数：
- `topic`（必填）
- `target_journal`（可选）
- `baseline_models`（可选）
- `improvement_modules`（可选）
- `key_results`（可选）
- `report_summaries`（可选）
- `constraints`（可选）
- `candidate_title_count`（默认 6，范围 3-10）

## 5.11 `init_model_diagram_pack`

作用：
- 一键生成“模型结构图生产包”，用于你说的稳定流程：
  - 先定真值拓扑
  - 再用 Nano Banana Pro 风格化
  - 多轮只改样式
  - 最后做局部放大图

参数：
- `project_dir`（必填）
- `model_name`（必填）
- `drawio_file_path`（可选，推荐；提供后作为唯一真值源）
- `truth_priority`（`auto`/`drawio`/`mermaid`，默认 `auto`）
- `mermaid_code`（可选，提供则作为真值拓扑）
- `modules`（可选，未提供则用默认模块）
- `output_subdir`（默认 `figures/model-diagram`）
- `force`（默认 `false`）

输出文件：
- `01-topology-truth.drawio`（使用 draw.io 真值时）
- `01-topology-lock.json`（使用 draw.io 真值时）
- `01-topology-truth.mmd`
- `02-nanobanana-main-prompt.txt`
- `03-nanobanana-style-refine-prompt.txt`
- `04-nanobanana-zoom-prompts.md`
- `05-integrity-checklist.md`
- `latex-figure-snippet.tex`

## 6. `.olignore` 最佳实践

推荐只屏蔽编译产物，不要屏蔽正文源文件。

推荐模板：

```text
*.aux
*.bbl
*.blg
*.fdb_latexmk
*.fls
*.log
*.spl
*.out
*.toc
*.synctex.gz
*.zip
*.pdf
```

避免写入以下规则（除非你明确要禁传）：
- `sections/*`
- `references.bib`

## 7. 常见问题排查

## 7.1 `ols: command not found`

原因：脚本目录不在 PATH，或 venv 未激活。  
处理：
- 激活 venv：`source /root/overleaf-ce-mcp/.venv/bin/activate`
- 或显式执行 `ols` 绝对路径

## 7.2 `ImportError: QtCore`（Windows 常见）

原因：`ols login` 依赖 GUI 组件。  
处理：在无 GUI 环境使用手工 `.olauth` 方案。

## 7.3 访问 `/project` 被重定向到 `/login`

原因：cookie 无效或过期。  
处理：重新获取浏览器 cookie 并更新 `.olauth`。

## 7.4 编译返回 200 但仍失败

这是正常现象，需看 `compile_status`：
- `status_code=200` 不代表成功
- 以 `compile_status == success` 为准

## 7.5 replace 模式删不净子目录文件

已修复：支持多级路径删除。若旧环境仍异常，执行：
- `apply_compat_patches`
- 重新同步

## 7.6 误把整个 workspace 同步到单项目

已增加护栏自动收敛路径。仍建议你：
- 明确设置 `project_name`
- 工作区保持“一个项目一个子目录”结构

## 8. Ocean Engineering OA 推荐实战流程

1. 用模板初始化目录。  
2. 用 draw.io MCP 先确定模型结构真值图（`.drawio`）。  
3. 用 `init_model_diagram_pack` + `drawio_file_path` 生成 Nano Banana Pro 提示词包。  
4. 用 `generate_deep_research_prompt_set(round_stage=r1)` 生成第一轮多组提示词并在 GPT 网页版运行。  
5. 用 `ingest_deep_research_report` 解析 R1 报告，提取 DOI/arXiv/BibTeX 草稿。  
6. 用 `build_related_work_pack` 拉取候选论文与补充 BibTeX 草稿。  
7. 如仍有不确定点，用 `generate_deep_research_prompt_set(round_stage=r2)` 做二轮深挖，并再次解析。  
8. 用 `synthesize_paper_strategy` 输出题目候选、创新点和写作侧重点。  
9. 将 BibTeX 合并到 `references.bib`，把报告摘要与 `note_markdown` 作为引言/相关工作素材。  
10. 先本地 `compile_latex`，确保主结构可编译。  
11. 用 `upload_project_dir` + `dry_run=true` 检查打包计划。  
12. 首次上传不传 `target_project`（创建新项目）。  
13. 后续迭代使用 `target_project + merge`。  
14. 需要远端完全对齐时，用 `target_project + replace`。  
15. 每次关键更新后执行 `health_check_project`。
